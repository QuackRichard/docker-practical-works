# docker-practical-works
Практические работы для студентов для освоения технологии Docker.
![Docker logo](https://upload.wikimedia.org/wikipedia/commons/7/79/Docker_%28container_engine%29_logo.png)

Docker - технология, позволяющая контейнизировать приложение, что обеспечит изолированное исполнение приложения. После этой формулировки можно перейти к такому описанию Docker-технологии, что бы не знакомый с этим человек понял его суть.
---
> Компания FishTech разрабатывает приложение, которое позволяет пользователям создавать, развивать и показывать свой аквариум другим пользователям. В компанию FishTech приходит новый сотрудник Иван, ему поручают разработать новую логику, по которой плавать рыбы, будут более реалистично в аквариуме пользователя. Лидер команды Ивана, сообщает ему, что бы начать разработку новой логики, Ивану нужно получить свою копию приложения FishTech, что бы своими изменениями не сломать приложение, которым **уже** пользуются реальные пользователи. Иван узнаёт у Андрея(лидер команды), как же ему уже приступить к написанию новой логики приложения как можно скорее. После чего он узнаёт, что компания использует Docker для разработки, тестирования и стресс-тестов своего приложения. После этого Иван получает свою копию приложения и начинает разработку. После создания новой логики и локального тестирования, Иван отправляет изменения на полное тестирование всего приложения с его новой логикой, всё тесты проходят успешно не только потому что Иван написал хороший код, а так же из-за того, что среда в которой он тестировал свой код, совпадала со средой главного приложения. Одинаковая среда позволила не думать о том, какие стороние эффект могут быть от библиотек разных версий основного приложения и локального с помощью которых пишется приложение FishTech.

1. Для чего нужен:
    - Что бы обеспечить кому угодно **одинаковое**, **предскауемое** и легко **воспроизводимое** приложение/сервис/инфраструктуру в один клик.
3. Для чего не нужен:
    - Для Desktop или GUI приложений.
    - Для массивных приложений, которым важно получать прямой доступ к железу.
    - Для приложений у которых будет только один экземпляр.
    - Для прототипов приложений.

Философия Docker: 1 процесс - 1 контейнер.

История контейнеризации:
<br>1970-e года: chroot - Системный вызов, который позволяет для какого-либо процесса задать временный, новый корневой каталог. Раньше использовалось, что бы создатавать тестовые сборки GNU/Linux дистрибутивов. Сейчас этот системный вызов чаще всего используется для востановления системы. Можно запустить **только** имея root права.
<br>2000-e года: jail - Механизм виртуализации в ОС FreeBSD, основывающаяся на системном вызове chroot. В отличие от chroot, jail уже может управлять сетью, заранее примонтированный каталог с [DevFS](https://www.opennet.ru/man.shtml?topic=devfs&category=5&russian=1). Особенность jail заключается в том, что у каждой "виртуальной ОС" должен быть свой уникальный для сети IP и hostname.
<br>2010-е года: Docker - Современный механизм управления контейнернерами, изначально базировавшийся на LXC, затем перешедший на свой собсвенный механизм контейнирезации containerd.

Термины:
1) Dockerfile - рецепт(чёткая инструкция), который позволят создать именно такой контейнер, который выполнит поставленную задачу, то есть настроит систему в контейнере определённым, заданым в этом файле образом.
2) Image - Созданный из Dockerfile'а контейнер, который можно сохранить, распространить или запустить, в случае чего Image станет Container'ор.
3) Container - То, во что превращается Image с помощью containerd.
4) Registry - Место хранения различных Image'ов
5) Docker Compose - Позволяет описать **ЧТО** нужно после запуска конетейнера, а не **КАК** это создать. Благодаря подходу **ЧТО** позволяет развернуть приложение, состоящие не только из одного сервиса.
